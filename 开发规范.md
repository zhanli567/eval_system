# 开发规范

## 基础技术栈

1. 后端使用 Spring Boot、JDK 21、Maven 3.9.x。
2. 数据库使用本地 MySQL，默认连接信息：`localhost:3306`，账号 `root`，密码 `123456`。
3. 后端数据访问使用 MyBatis / MyBatis Plus，不使用 Repository。
4. 前端使用 Vue 3 + TypeScript + Element Plus。
5. 前端使用 `vue-router` 管理页面路由。

## 开发流程

1. 新模块优先从 DDL 设计开始，先讨论字段和表结构，确认后再进入后端、前端实现。
2. 用户明确要求“先讨论”时，只讨论方案，不直接实现代码。
3. 涉及数据库结构时，DDL 文件统一维护在 `DDL` 文件夹中。
4. 每个业务模块的 DDL 独立维护，例如评估器模块使用 `DDL/03_eval_evaluator.sql`。
5. DDL 设计阶段只关注字段和注释，暂不考虑主键、非空、唯一、外键等约束。
6. 数据库字段尽量保持最小可用，不提前设计无明确需求的冗余字段。
7. `id` 字段统一使用字符串类型。
8. 时间字段统一使用字符串类型，存时间戳。
9. 删除标记优先使用 `is_deleted` 这类是否删除字段。
10. 不主动提交 commit，除非用户明确要求。
11. 需要提交时，按用户指定的 commit message 提交。
12. 需要推送远程时，只有用户明确要求后再执行。

## 后端目录规范

后端目录结构保持清晰：

- `controller`：控制层
- `service`：服务接口
- `service/impl`：服务实现类
- `mapper`：Mapper 接口
- `pojo`：数据库实体类
- `dto`：请求和响应对象
- `common`：通用响应、分页、异常处理等
- `config`：全局配置

## 后端编码规范

1. Controller 接口统一返回通用响应对象：
   - `code: Integer`
   - `msg: String`
   - `data: Object`
2. 简单单表 SQL 可以放在 Mapper 注解中。
3. 复杂 SQL 统一放到 `resources/mapper/*.xml`。
4. Service 层负责业务校验、数据规整、版本规则、参数提取等业务逻辑。
5. DTO 面向接口请求和响应，不直接复用数据库实体作为接口模型。
6. 发布版本只读，草稿版本允许修改。
7. 对于从文本中提取结构化信息的能力，后端必须兜底实现，前端只做交互增强。
8. 后端不要在服务启动时自动初始化或修改业务数据，除非用户明确要求。
9. 新增业务模块 Mapper 包后，需要确认启动类或配置中的 `@MapperScan` 覆盖到对应 mapper 包。
10. Mapper XML 的 `namespace`、方法名、返回映射字段需要与 Mapper 接口和 DTO 字段保持一致，尤其注意数据库下划线字段到 Java 驼峰字段的映射。
11. 批量新增、批量导入、全量覆盖等写操作需要放在事务中，业务校验失败时不写入部分数据。
12. 文件导入类能力由后端兜底校验表头、字段类型和必填项，前端只做上传、预检查和交互提示。

## 接口文档规范

1. 每新增或修改后端接口后，需要同步维护项目根目录的 `接口文档.md`。
2. 接口文档至少记录接口地址、请求方式、请求参数、响应字段和业务说明。

## 前端目录规范

前端推荐目录结构：

- `api`：接口请求
- `config`：模块配置、菜单配置等
- `layouts`：整体布局
- `router`：路由配置
- `views`：路由页面
- `modules/{moduleName}`：具体业务模块
- `modules/{moduleName}/components`：模块内组件
- `modules/{moduleName}/composables`：业务状态和操作逻辑
- `modules/{moduleName}/types`：模块内类型定义

## 前端编码规范

1. `App.vue` 尽量只保留应用入口和路由出口。
2. 页面采用模块化结构，不把业务逻辑集中写在 `App.vue`。
3. 页面风格参考阿里云百炼平台，整体采用左侧导航栏 + 右侧内容区布局。
4. 左侧导航栏中的一级模块应对应独立路由地址。
5. 弹窗、表格、表单等交互需要考虑数据量较多时的滚动、分页和布局稳定性。
6. 表单提交前做基础校验，后端仍保留完整校验。
7. 列表页支持分页、搜索、筛选等当前需求中明确的能力。
8. 对于参数、字段、版本这类动态列表，前端展示应保持稳定宽度，避免输入时布局跳动。
9. 新增、编辑、删除、导入、覆盖成功后，需要刷新相关列表、详情或同步更新本地状态，避免界面残留旧数据。
10. 路由页面负责组织页面结构，复杂交互拆到模块组件和 composable 中，避免单文件承载过多业务逻辑。
11. 文件上传和批量操作需要提供上传中、成功、失败等明确反馈，失败时不要静默保留错误状态。

## 前后端字段规范

1. 枚举类字段使用稳定的英文 code 存储和传输，前端负责展示中文名称，例如 `category`、`boolean`、`number`、`text`。
2. 对于创建后不允许修改的字段，前端编辑态应禁用，后端也必须做兜底校验。
3. 动态字段、参数、表头等可排序配置统一维护排序字段，例如 `display_order` / `displayOrder`，读写时都按该顺序处理。

## 数据与版本规则

1. 自定义业务对象如评估器、评测集等，新增后默认生成草稿版本。
2. 草稿版本可以修改。
3. 发布版本不允许修改。
4. 发布时生成新的只读版本，并更新主表的最新版本指向。
5. 复制、基于预置创建等能力只作为创建入口的快捷方式，不额外设计无明确需求的来源字段。
6. 参数、字段等配置要贴合最终使用场景，不只服务当前页面展示。

## DDL 与数据维护

1. DDL 文件只放建表语句，除非用户明确要求加入初始化数据。
2. Mock 数据、迁移数据、修复数据应单独执行，确认后不要混入建表 DDL。
3. 对已有数据库做删表、重建、导入前，需要先确认表和数据量。
4. 执行数据重建前应尽量备份相关表。
5. 避免通过 PowerShell 管道导入中文 SQL；执行中文 SQL 时优先使用 MySQL 客户端 `SOURCE` 或确保 `--default-character-set=utf8mb4`。
6. 本地数据库修复完成后，要校验数据量、关键字段、重复数据和乱码风险。
7. 如果项目启用了 `backend/src/main/resources/schema.sql` 作为本地启动建表脚本，业务 DDL 确认后需要同步维护 `DDL/*.sql` 和 `schema.sql`，避免本地启动环境与正式 DDL 文件不一致。
8. 字段 `COMMENT` 保持简洁，只描述业务含义，不重复书写主键、非空、唯一等约束信息。

## 验证规范

每次修改后尽量执行基础验证：

- 后端：`mvn test`
- 前端：`npm run build`
- Git：确认 `git status` 只包含预期变更

补充要求：

1. 验证服务能运行后，如果不是明确要求保持运行，验证完应停止。
2. 本地调试产生的日志文件、`logs` 目录等不要保留在工作目录中。
3. 如果只修改后端或只修改前端，可按影响范围选择验证命令，但最终需要说明哪些验证执行了、哪些没有执行。
4. 前端构建中的 chunk 体积提示属于构建警告，不等同于构建失败。

## Git 规范

1. 不主动提交 commit。
2. 用户要求提交时，先确认工作区只包含预期变更。
3. 不回滚用户或其他流程产生的未关联改动。
4. 提交信息使用用户指定内容。
5. 推送远程必须在用户明确要求后执行。
6. 提交后检查 `git status`，确保工作区干净或只剩用户自己的未跟踪文件。
7. 提交时只暂存本次任务相关文件，其他未跟踪或未完成模块文件保持原状。

## 协作注意事项

1. 遇到用户已有改动，默认认为是用户需要保留的内容。
2. 不使用 `git reset --hard`、`git checkout --` 等破坏性命令，除非用户明确要求。
3. 修改文件时保持改动范围收敛，不做无关重构。
4. 实现时优先遵循项目已有风格，而不是引入新的抽象或工具。
5. 对业务不明确的字段，不提前扩展。
6. 对会影响后续模块的设计，要优先从最终使用场景倒推字段和接口。
